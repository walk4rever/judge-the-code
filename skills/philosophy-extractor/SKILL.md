# Philosophy Extractor — 设计哲学提炼器

提炼代码库的隐含设计信仰：哪里精妙、哪里合理、哪里存疑。

## 核心理念

代码库不只是功能的集合，它是一系列**设计决策**的沉淀。
这些决策透露了作者的品味、经验、以及在特定约束下的权衡取舍。

读懂这些决策，比单纯看懂代码更有价值——
因为你学到的不是"这里写了什么"，而是"为什么这样写，还有什么别的选择"。

## 调用方式

```
/philosophy-extractor <项目路径>
/philosophy-extractor .
/philosophy-extractor ~/projects/some-repo
```

---

## Token 使用原则

> **核心策略：有目的地采样源码，找模式而非读实现。**

1. **优先复用** `UNDERSTANDING.md`：如果 understand-repo 已运行过，直接读取，跳过重复分析
2. **Grep 先于 Read**：用关键词定位典型代码片段，不要盲目读整个文件
3. **采样而非全读**：每个维度只读 3-5 个代表性文件，每文件最多 80 行
4. **测试文件是宝藏**：这里 **允许** 读测试文件（understand-repo 明确禁止），测试命名和结构最能透露设计取向
5. **禁止读取**：生成文件（`dist/`, `build/`）、依赖包（`node_modules/`, `vendor/`）、二进制文件

## 执行流程

### Phase 0：准备上下文

1. **输入验证**（路径异常时立即停止并提示）：
   - 路径为空 → `❌ 请提供项目路径，例如：/philosophy-extractor .`
   - 路径不存在 → `❌ 路径 {path} 不存在`
   - 路径是文件 → `❌ {path} 是文件，请提供项目根目录`

2. **检查是否有 `UNDERSTANDING.md`**（understand-repo 的输出）：

   ```bash
   # 检查项目根目录
   ls UNDERSTANDING.md 2>/dev/null
   ```

   - **存在** → 读取全文作为背景上下文（**直接跳到 Phase 1，跳过技术栈重新分析**）
   - **不存在** → 快速读取 `package.json` / `go.mod` / `pyproject.toml` 等识别语言和框架（最多读 2 个文件，各 50 行），然后进入 Phase 1

3. 输出：
   ```
   🔍 开始提炼设计哲学，启动 4 个分析 Agent...

     Agent 1 — 命名与抽象哲学    ⏳ 分析中
     Agent 2 — 错误处理取向      ⏳ 分析中
     Agent 3 — 测试与质量信仰    ⏳ 分析中
     Agent 4 — 架构决策考古      ⏳ 分析中

   预计完成时间：3-5 分钟
   ```

### Phase 1：并行 4 Agent 分析

**同时启动以下 4 个 Explore 子 Agent，全部并行执行：**

---

#### Agent 1 — 命名与抽象哲学分析师

**任务**：通过命名风格和抽象层级，揭示代码库的表达取向。

**步骤**：

1. 用 Glob 找 5-8 个核心源码文件（排除测试文件、生成文件）：
   - 优先选 `services/`, `controllers/`, `handlers/`, `core/`, `domain/` 下的文件
   - 各读前 **60 行**

2. 分析**命名风格**，寻找以下模式：

   | 模式 | 示例 | 透露的取向 |
   |------|------|-----------|
   | 动词驱动 | `processOrder()`, `validateUser()` | 行为优先，过程式思维 |
   | 名词驱动 | `OrderProcessor`, `UserValidator` | 对象优先，OOP 思维 |
   | 业务语言 | `chargeSubscription()`, `renewLease()` | 领域驱动，DDD 倾向 |
   | 技术语言 | `executeQuery()`, `parseJson()` | 实现优先，技术驱动 |
   | 极简缩写 | `usr`, `cfg`, `ctx` | 简洁优先，可能牺牲可读性 |
   | 过度冗长 | `UserAccountInformationService` | 防御性命名，可能过度设计 |

3. 分析**抽象层级**：

   - 接口/抽象类的数量 vs 具体实现数量（Grep `interface|abstract|Protocol|trait`）
   - 是否有过度抽象的迹象（5层继承链、每个功能都有接口）
   - 是否有欠抽象的迹象（大量重复代码、超长函数）

4. 分析**函数/方法规模**（Grep 函数定义，估算平均长度）：
   - 短函数倾向（< 20行）→ 单一职责意识强
   - 长函数倾向（> 100行）→ 可能职责混杂

**输出格式**：
```
## 命名与抽象哲学

### 命名取向
[动词驱动 / 名词驱动 / 业务语言优先 / 技术语言优先]
**证据**：
- `getUserByEmail()` → 动词驱动，行为清晰
- `EmailLookupService` → 也有名词形式，混合风格
**结论**：该项目倾向 [xxx]，原因是...

### 抽象层级
[过度抽象 / 适度抽象 / 扁平直接]
**证据**：
- 发现 3 个 interface，12 个 concrete class → 抽象比例适中
- 最深继承链：3 层（可接受）
**结论**：...

### 函数规模
[短函数风格 / 中等 / 长函数风格]
**证据**：采样的 8 个文件中，函数平均约 [N] 行
**结论**：...

### 值得关注的命名决策
- [具体例子 + 分析]
```

**Agent 1 完成后，输出：**
```
✅ Agent 1/4 完成 — 命名与抽象哲学已提炼
```

#### Agent 2 — 错误处理取向分析师

**任务**：通过错误处理方式，揭示代码库对"意外"的态度。

**步骤**：

1. Grep 搜索错误处理关键词（跨语言）：
   ```
   try|catch|throw|except|raise|panic|recover|Result|Either|Option|unwrap|?
   err != nil|if err|.error()|reject|Promise.reject
   ```
   定位 5-10 处典型错误处理代码，读取上下文各 **20 行**

2. 分析**错误处理风格**：

   | 风格 | 特征 | 透露的取向 |
   |------|------|-----------|
   | 快速失败 | `panic`, `assert`, `throw` 用于不可恢复错误 | 防御性强，宁可崩溃也不带病运行 |
   | 防御包裹 | 大量 `try/catch`，返回默认值 | 容错优先，可能掩盖问题 |
   | 显式传播 | `Result<T, E>`, `if err != nil` | 强制调用方处理，类型安全 |
   | 静默忽略 | `catch (e) {}`, `_ = err` | 可能的隐患，也可能是有意为之 |
   | 统一处理 | 全局 error handler，middleware 层统一捕获 | 架构意识强 |

3. 检查**错误信息质量**：
   - 错误消息是否包含上下文（`"failed to create user: email already exists"` vs `"error"`）
   - 是否有错误码体系（HTTP 状态码使用是否准确）
   - 是否有结构化错误类型（自定义 Error 类）

4. 检查**边界保护意识**：
   - Grep `null|undefined|nil|None` 检查空值处理
   - 是否有输入校验层（zod/joi/pydantic/class-validator）
   - 校验是否在入口处统一，还是散落各处

**输出格式**：
```
## 错误处理取向

### 主导风格
[快速失败 / 防御包裹 / 显式传播 / 混合]
**证据**：
- `src/services/UserService.ts:45` — throw 用于不合法的业务状态（快速失败）
- `src/api/handler.ts:12` — 全局 catch 统一处理（架构意识）
**结论**：...

### 错误信息质量
[信息丰富 / 一般 / 过于简略]
**证据**：...

### 边界保护
[严格 / 一般 / 宽松]
**证据**：
- 使用 zod 在 API 层统一校验 → 边界清晰
- 但 service 层未做二次校验 → 信任上游，有一定风险
**结论**：...

### 值得关注的错误处理决策
- [具体例子 + 分析]
```

**Agent 2 完成后，输出：**
```
✅ Agent 2/4 完成 — 错误处理取向已分析
```

#### Agent 3 — 测试与质量信仰分析师

**任务**：通过测试文件（这里允许读）揭示项目对"质量"的态度。

**步骤**：

1. 用 Glob 找测试文件：
   ```
   **/*.test.ts, **/*.spec.ts, **/*_test.go, **/test_*.py, **/*.test.js
   ```
   挑选 3-5 个代表性测试文件，读前 **80 行**

2. 分析**测试类型分布**：

   | 测试类型 | 识别特征 | 透露的取向 |
   |---------|---------|-----------|
   | 纯单元测试 | 大量 mock，依赖全部替换 | 隔离优先，可能过度 mock |
   | 集成测试 | 真实数据库/服务，少量 mock | 真实性优先 |
   | E2E 测试 | `playwright`, `cypress`, `supertest` | 用户视角，端到端保障 |
   | 快照测试 | `toMatchSnapshot()` | 防回归优先，可能测试脆弱 |
   | 属性测试 | `fast-check`, `hypothesis` | 数学严谨性，边界值意识强 |

3. 分析**测试命名风格**（最能透露测试哲学）：

   | 命名风格 | 示例 | 透露的取向 |
   |---------|------|-----------|
   | 行为描述 | `"should return 404 when user not found"` | BDD 风格，可读性优先 |
   | 功能描述 | `"getUser_returnsNull_whenNotFound"` | 结构化，可能过于技术 |
   | 极简 | `"test getUserById"` | 不重视测试文档化 |
   | Given-When-Then | 注释或结构清晰 | 意图明确，维护友好 |

4. 分析**mock 策略**：
   - Grep `mock|stub|spy|jest.fn|unittest.mock|gomock`
   - mock 比例高 → 单元测试纯度高，但可能测试的是 mock 而非真实逻辑
   - mock 比例低 → 集成测试倾向，更真实但更慢

5. 检查**测试覆盖边界**：
   - 是否有边界条件测试（空列表、最大值、并发场景）
   - 是否只测"happy path"

**输出格式**：
```
## 测试与质量信仰

### 测试类型分布
[单元测试主导 / 集成测试主导 / 混合 / 测试覆盖薄弱]
**证据**：
- 发现 24 个 `.test.ts` 文件，其中 18 个使用 mock → 单元测试主导
- 发现 2 个 e2e 测试文件 → 有端到端意识但覆盖有限
**结论**：...

### 测试命名风格
[BDD 行为描述 / 功能描述 / 混合 / 不规范]
**证据**：
- `"should reject invalid email format"` → BDD 风格（清晰）
- `"test_user_creation"` → 功能描述（一般）
**结论**：...

### Mock 策略
[过度 mock / 适度 mock / 轻 mock]
**结论**：...

### 测试质量亮点
- [值得学习的测试写法]

### 测试质量隐患
- [可能导致问题的测试取向]
```

**Agent 3 完成后，输出：**
```
✅ Agent 3/4 完成 — 测试与质量信仰已分析
```

#### Agent 4 — 架构决策考古师

**任务**：挖掘"非显而易见"的架构决策，解释背后的权衡。

**步骤**：

1. **读 git 提交历史**（如果是 git 仓库）：
   ```bash
   git log --oneline -30
   git log --oneline --follow -- [核心文件] | head -10
   ```
   从提交信息中推断：重大重构、技术选型变化、设计理念演进

2. **分析依赖选择的"为什么"**（读 `UNDERSTANDING.md` 中依赖部分，或重新读 `package.json`）：

   重点关注**非显然的选择**：
   - 同类库中选了哪个？（用了 `drizzle` 而非 `prisma`？用了 `zod` 而非 `joi`？）
   - 是否自建轮子？（说明什么场景下找不到满意的开源方案）
   - 是否有明显的"奥卡姆剃刀"风格（最小依赖）或"拿来主义"（大量第三方库）

3. **分析配置文件透露的工程态度**：

   | 文件 | 关注点 |
   |------|--------|
   | `.eslintrc` / `biome.json` | 规则严格程度，团队对代码风格的在意程度 |
   | `tsconfig.json` | `strict: true` 的使用（类型安全意识）|
   | `Dockerfile` | 多阶段构建？最小镜像？安全意识？ |
   | `.github/workflows/` | CI 严格程度，部署自动化水平 |
   | `commitlint` / `husky` | 提交规范意识 |

4. **识别"反常规"决策**（最有价值的部分）：

   与业界主流做法不同的地方，可能是：
   - 刻意的权衡（有充分理由）
   - 历史包袱（当初有理由，现在可以改了）
   - 个人偏好（无对错，但值得注意）

   判断方法：如果一个决策"让人意外"，它就值得被解释。

**输出格式**：
```
## 架构决策考古

### 技术选型取向
[最小依赖主义 / 生态整合主义 / 自建轮子倾向]
**证据**：
- 用 drizzle 而非 prisma → 更轻量，更接近 SQL 原语
- 未引入 react-query，用原生 fetch → 简单场景不过度封装
**结论**：该项目倾向轻量直接，警惕过度抽象

### 工程规范意识
[严格 / 一般 / 宽松]
**证据**：
- `tsconfig.json` 开启 `strict: true` → 类型安全意识强
- 有 commitlint 配置 → 提交规范意识
- 无 Dockerfile → 部署尚未容器化
**结论**：...

### 非显然的关键决策

| 决策 | 常规做法 | 项目选择 | 可能的理由 |
|------|---------|---------|-----------|
| ORM | Prisma | Drizzle | 更轻量，类型推导更精准 |
| 状态管理 | Redux | Zustand | 简单场景，避免样板代码 |
| ... | ... | ... | ... |

### Git 历史透露的设计演进
[如果 git log 有信息的话]
- [提交模式分析]
```

**Agent 4 完成后，输出：**
```
✅ Agent 4/4 完成 — 架构决策已考古

所有 Agent 完成，正在综合提炼设计信仰...
```

### Phase 2：综合输出

收集所有 4 个 Agent 的结果后，进行两步综合：

#### Step 1：判定每条决策的评价

对分析中发现的每一个值得记录的设计决策，打上标签：

| 标签 | 含义 | 判断标准 |
|------|------|---------|
| 🔮 **精妙** | 非显而易见的好决策，值得主动学习 | 解决了一个真实问题，且比常规做法更优雅 |
| ✅ **合理** | 标准做法，没什么特别，但没问题 | 业界常见，符合场景需求 |
| ⚠️ **存疑** | 有一定风险或代价，值得讨论 | 有明显的 tradeoff，且代价可能被忽视 |
| ❌ **反模式** | 有明确危害，应当避免或改进 | 违反安全、性能或可维护性的基本原则 |

**打标签原则**：
- 同一个决策在不同场景下评价可能不同，要结合项目背景
- 不确定时用 ✅，宁可保守也不随意打 ❌
- 每条必须有**具体证据**（文件路径 + 代码片段），不能只有结论

### Phase 3：深度问答模式

生成完 `PHILOSOPHY.md` 后，进入**设计讨论模式**。

**处理用户追问时的原则：**

1. **引用具体证据**：每个判断都要有文件路径 + 行号，不发表无根据的观点
2. **比较视角**：讨论一个决策时，主动对比"如果换一种做法会怎样"
3. **承认不确定性**：遇到"不清楚为什么这样做"的情况，直接说，不编理由
4. **追问驱动**：每次回答后，提供 1-2 个"延伸问题"引导深入

**深挖触发词**：
- "展开说说" / "为什么" → 加载更多相关代码，深入分析
- "跟 [另一个项目] 比" → 进行横向对比分析
- "怎么改" → 给出具体的改进建议（不是泛泛而谈）
- "教我用这个模式" → 提炼出可复用的代码模板

---

## 分析质量标准

- **每条评价必须有证据**：文件路径 + 代码片段，不能凭感觉
- **区分"设计决策"和"代码风格"**：`camelCase vs snake_case` 是风格，`是否使用 Result 类型` 是设计决策
- **精妙的决策不超过 5 条**：如果超过，重新筛选——真正精妙的决策很少见
- **反模式要谨慎**：不确定是否有害时，降级为存疑，避免误判
- **结合项目类型**：CLI 工具和大型后端服务的"合理决策"标准不同
